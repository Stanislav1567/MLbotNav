# ТЗ: Ускорение прогона + безопасный CPU-контроль (без рывков)

Дата: 2026-05-22  
Проект: `MLbotNav`

## 1. Цель
- Ускорить тяжелые прогоны (`adaptive_auto_train`), особенно первый этап поиска.
- Не перегружать CPU выше безопасного уровня.
- Не реагировать на кратковременные (1-2 сек) пики; управлять нагрузкой по устойчивому окну.

## 2. Проблема
- Первый этап (`search_gate_candidate`) может долго идти при большой сетке.
- Общий CPU может быть низким из-за локального bottleneck, а не из-за простоя всей системы.
- Текущий контроль реагирует на одиночные замеры и не учитывает устойчивость нагрузки.

## 3. Что должно быть реализовано

### 3.1 Устойчивый CPU-контроль по окну
- Замер CPU не точкой, а окном (`window_sec`, по умолчанию 10 сек).
- На основании среднего/максимального CPU за окно:
  - если устойчиво горячо (`>= hot_threshold`) — снижать threads,
  - если устойчиво холодно (`<= cool_threshold`) — повышать threads,
  - иначе держать текущий уровень.
- Цель: убрать дергание из-за секундных всплесков.

### 3.2 Профили скорости
- Добавить профили запуска:
  - `custom` — как задано пользователем.
  - `turbo` — агрессивное ускорение (узкая сетка, меньше repeats).
  - `balanced` — средний компромисс.
  - `full` — широкий поиск.

### 3.3 Раздельные контуры long/short
- Профили и CPU-контроль должны одинаково работать для `long_only` и `short_only`.
- Ключ памяти должен оставаться изолированным по `signal_mode`.

## 4. Новые параметры CLI
- `--speed-profile {custom,turbo,balanced,full}`
- `--cpu-control-window-sec` (default `10`)
- `--cpu-sample-interval-sec` (default `1`)
- `--cpu-hot-threshold-pct` (default `90`)
- `--cpu-cool-threshold-pct` (default `45`)

## 5. Ограничения безопасности
- `cpu_max_pct` сохраняется как верхний лимит.
- Никакие внешние/чужие прогоны не останавливаются.
- Только внутреннее управление threads текущего запуска.

## 6. Критерии приемки
1. На `turbo` запуск становится заметно быстрее относительно `custom/full`.
2. При кратком пике CPU нет мгновенного “пиления” threads.
3. Если CPU держится высоко > окна, threads понижаются автоматически.
4. Если CPU стабильно низкий, threads повышаются автоматически.
5. Все параметры и замеры окна пишутся в `summary.history[*]`.
