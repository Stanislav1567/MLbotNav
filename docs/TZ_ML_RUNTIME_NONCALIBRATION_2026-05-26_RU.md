# ТЗ: ML Runtime Without Calibration (v1)

Дата: 2026-05-26  
Статус: ACTIVE (R8 закрыт, режим поддержки консистентности)  
Назначение: единый source of truth для ML-runtime без Optuna/AKFP-калибровки.

## 1. Scope
1. Входит:
   - ingestion/data quality/dataset core;
   - feature pipeline;
   - train/eval/backtest;
   - inference/runtime;
   - execution parity;
   - registry/gates/monitoring;
   - table-canon/audit chain.
2. Не входит:
   - калибровочный контур (`AKFP/*`, `APTuna/*`, `optuna_*`, `akfp_*`);
   - отдельный runtime-блок для внешних потребителей.

## 3. Архитектурные правила
1. Runtime-контур единственный и канонический.
2. Long/short split обязателен, смешивание запрещено.
3. Вход/выход в `exchange_like` режиме воспроизводим по артефактам.
4. Gate-chain обязателен: `P23 -> P24 -> P26`.
5. Любой релизный promote только при полном PASS по обязательным гейтам.

## 6. Правила входа/выхода сделки (runtime)
1. Канонический вход: исполнение на `t+1` (следующая свеча), без look-ahead.
2. Поддерживаемые execution path:
   - `order_type=market`
   - `order_type=limit` (только при подтвержденной parity-проверке)
3. Stop/TP правила применяются одинаково для backtest/inference/execution trace.
4. `NO_TRADE` — нормальный исход при провале risk-gate.

## 7. Risk-gates
1. `min_tp_reach_prob` обязателен и должен быть протянут сквозь:
   - daily cycle -> adaptive -> grid/optuna search -> backtest.
2. `min_expected_move_pct` в канонических единицах (доля, `0.01 = 1%`).
3. SLA inference в release-контуре — hard gate.
4. Stress contour (`5/10/15 bps`) обязателен для финального audit пакета.

## 11. Definition of Done
1. Все runtime-сигналы соответствуют единому контракту.
2. `signal -> entry -> exit -> pnl` воспроизводимы по логам и аудитам.
3. Long/short split соблюдается без смешивания.
4. Promote невозможен без полного `gate/stage PASS`.
5. Все `PARTIAL/TODO` по ML-runtime без калибровки переведены в `DONE`.

## 12. Жесткое правило идентичности с внешним калибровочным контуром
1. Основной ML-runtime контур — канонический источник истины для параметров, фич, гипотез, входов/выходов, таблиц и audit/gate правил.
2. Внешний контур (включая Optum/Optuna-потребителя) не имеет права заводить отдельный runtime-блок внутри этого ML-контура.
3. Внешний контур может только читать данные/артефакты и предлагать кандидатные профили.
4. Любое изменение сначала вносится в основной ML-runtime, затем фиксируется в хронологии.
5. Любое расхождение считается дефектом интеграции и исправляется через основной контур, не через временный fork.
