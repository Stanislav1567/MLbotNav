# TZ: Stable Long/Short + Execution Parity + Honest Backtest (v1)

Дата: 2026-05-26  
Статус: ACTIVE  
Источник: консолидировано из рабочих сообщений и проверок кода за 2026-05-26.

## 1. Цель
1. Убрать путаницу между ML-контуром, калибровочным контуром и execution-контуром.
2. Зафиксировать единые правила long/short (раздельно), входа/выхода, проверки качества и релизных gate.
3. Сделать быстрый старт проверки на 1 дне, затем масштабирование на 30/60/120 дней.

## 2. Жесткие требования (MUST)
1. Long и short ведутся как два независимых контура, без смешивания.
2. Сигнал на свече `t`, вход не раньше следующей свечи `t+1`.
3. Текущий приоритет исполнения: `exchange_like` + `market`.
4. Выход только по формальным правилам стратегии, без "лучшей цены задним числом".
5. Любой вывод о качестве — только с OOS-проверкой, не по одному красивому прогону.
6. После каждого шага обновлять `docs/ACTIVE_WORK_ITEMS_RU.md` и `docs/CHANGELOG_CHRONOLOGY_RU.md`.

## 3. Что подтверждено по текущему коду
1. В `exchange_like` вход действительно идет на `t+1` (`entry_idx = i + 1`).
2. Текущие причины выхода: `tp`, `sl`, `sl_ambiguous`, `timeout`.
3. `dynamic_tp` сейчас вычисляется при входе как уровень, но лестницы перестановок внутри сделки пока нет.
4. В режиме `research` есть риск look-ahead (используется `future_return`), поэтому для parity/принятия в прод `research` не использовать как финальный.
5. Текущий модельный slippage в пайплайне: `5 bps` (настраиваемый), это модель издержек, а не параметр биржи.

## 4. Негативные риски, которые закрываем
1. Переобучение из-за подбора на коротком окне без walk-forward.
2. Иллюзия доходности при заниженных издержках исполнения.
3. Расхождение между backtest и реальным исполнением.
4. Потеря контекста при работе в нескольких чатах.

## 5. Execution Parity v1 (обязательный стандарт)
1. Сигнал фиксируется на закрытии `t`; сделка открывается не раньше открытия `t+1`.
2. Для market-входа цена входа = `next_open` + модельный slippage.
3. Выход допускается только по: `tp`, `sl`, `timeout` (+ `sl_ambiguous` как консервативное правило OHLC).
4. При одновременном касании SL/TP в одной свече используется консервативный порядок (SL-first), пока нет тиковых данных.
5. Любая сделка должна быть воспроизводима по логу: signal_time -> entry_time -> exit_time -> exit_reason -> pnl.

## 6. Политика slippage/издержек
1. Slippage обязателен даже для market (биржа не дает фиксированного параметра, но фактический слипп есть всегда).
2. Минимум три режима проверки:
1. Base: `5 bps`.
2. Stress-1: `10 bps`.
3. Stress-2: `15 bps`.
3. Стратегия не принимается, если edge исчезает в стресс-издержках.

## 7. Политика фич и гипотез (без комбинаторного взрыва)
1. Не делать brute-force "все со всем".
2. Использовать staged-подход:
1. Coarse: дешёвый широкий охват профилей/гипотез.
2. Refine: детальная донастройка top-K кандидатов.
3. Stability: проверка устойчивости на независимых окнах.
3. Выбор победителя по устойчивости (median/p10/DD/gate pass rate), а не по единичному максимуму.

## 7.1. Optuna: условное пространство (ОБЯЗАТЕЛЬНО)
1. Калибровка запускается одним поиском, где одновременно выбирается:
1. какие блоки `ON/OFF`;
2. какие параметры внутри включенных блоков тюнятся.
2. Формат логики:
1. `use_<block_name> = true/false`;
2. если `true` -> параметры блока доступны в trial;
3. если `false` -> блок принудительно зануляется и не влияет на score.
3. То же правило для гипотез:
1. `use_<hypothesis_name> = true/false`;
2. активные гипотезы участвуют в расчете, неактивные исключаются.
4. Это фиксирует цель: Optuna сама находит
1. лучший набор блоков,
2. лучшие параметры,
3. какие блоки/гипотезы лучше отключить.

## 7.2. Ограничения для условного поиска (ОБЯЗАТЕЛЬНО)
1. Минимум один блок всегда включен.
2. Long и short тюнятся строго отдельно.
3. В objective добавляется штраф за сложность (лишние активные блоки/параметры).
4. Победитель выбирается по OOS-устойчивости, а не по пику на одном окне.
5. Обязательный staged-режим:
1. Stage A: 1-day sanity (дешево, быстрый отсев).
2. Stage B: 30/60 дней (устойчивость).
3. Stage C: финальный контроль long/short в отдельных контурах.

## 8. План этапов
### P17.1 (быстрый sanity, 1 день)
1. Раздельный прогон `long_only` и `short_only` в `exchange_like`, `order_type=market`.
2. Проверка parity: вход `t+1`, корректные `exit_reason`, сходимость pnl из трейд-лога.
3. Сравнение slippage режимов `5/10/15 bps`.

Критерий PASS:
1. Нет нарушений времени входа.
2. Нет невалидных причин выхода.
3. Метрики не разваливаются полностью при stress-издержках.

### P17.2 (контур устойчивости)
1. Walk-forward минимум на 3 независимых окнах (long и short отдельно).
2. Фиксация gate-метрик для принятия/браковки.

### P17.3 (доработки исполнения)
1. Добавить событийный лог динамического TP (лестница TP updates).
2. Добавить явный `signal_exit` (если включаем его в стратегию).
3. Подготовить tick-aware режим parity (v2), если доступны данные.

## 9. Gate-критерии принятия (минимум v1)
### LONG
1. OOS положительный в большинстве окон (>=70%).
2. Drawdown в допустимом лимите.
3. Положительное expectancy после комиссий и slippage.

### SHORT
1. OOS положительный в большинстве окон (>=75%).
2. Более строгий лимит просадки.
3. Отдельная стресс-проверка на squeeze.

## 17. Контракт Optuna <-> ML (обязательный)
1. Optuna-контур и ML-контур — разные роли:
1. Optuna: офлайн-калибровка стратегии/параметров;
2. ML: обучение и применение модели на уже утвержденном профиле стратегии.
2. Запрещено в одном боевом рантайме:
1. одновременно крутить Optuna-поиск;
2. и принимать торговые решения как production ML-контур.
3. Поток данных между контурами:
1. Optuna публикует `strategy_profile_vN` (long и short отдельно);
2. профиль проходит OOS/gate/parity;
3. только после PASS профиль переводится в `active_strategy_profile`;
4. ML-контур читает только `active_strategy_profile`.
4. Разделение режимов:
1. Calibration mode: Optuna ON, ML runtime OFF.
2. Training/runtime mode: Optuna OFF, ML ON.
5. Версионирование:
1. каждый профиль имеет версию, окно train/test, метрики, решение gate;
2. без версии и записи в хронологию профиль не допускается в active.

## 18. CPU-политика калибровки (обязательная)
1. Для калибровочного контура целевая загрузка CPU: до `85%` (стабильно, без перегруза).
2. Если наблюдается недогруз (например 40-50%), допускается поэтапное увеличение `threads/search_workers`.
3. Допустимый коридор разгона: примерно `+30..35%` мощности от текущего профиля, при условии сохранения стабильности.
4. Любой разгон фиксируется в хронологии вместе с фактическим эффектом на скорость и стабильность.

## 19. Cross-Contour Schema Contract (SKV/Excel/MATB)
1. For Optuna calibration we enforce one mandatory column contract across:
1. SKV contour;
2. Excel tables/formulas;
3. MATB/Bot Navi main parser/validator.
2. Any new parameter/column is allowed only after synchronized update of all three layers.
3. Required order:
1. schema contract;
2. SKV update;
3. Excel update;
4. MATB parser/validator update;
5. only then calibration run.
4. Minimal control after each change:
1. Excel opens without errors and formulas stay valid;
2. canonical CSV/XLSX contains the new column in expected order;
3. gate audit shows no name/type conflicts.
5. Without a schema-sync record in `ACTIVE` and `CHANGELOG`, the change is considered incomplete.
