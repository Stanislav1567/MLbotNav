## 1. Цель

Довести контур minute-first до состояния, где:
1. freeze/readiness не обходится;
2. авто-цикл берет только качественные сетапы по целям;
3. состояние readiness корректно восстанавливается;
4. логика запуска стабильна и диагностируема;
5. память плохих сетапов не загрязняет другие окна/режимы.

## 2. Обязательные фиксы

### F1. Запрет обхода freeze-gate

Проблема: авто-скрипт сам поднимает `project_ready=true`, что обходит P0 freeze.

Требование:
1. Убрать автоматический unlock из `adaptive_auto_train.py`.
2. По умолчанию запускать цикл только если readiness разрешает train/search.
3. Если readiness блокирует, завершать запуск с явной причиной.

Критерий приемки:
1. При `project_ready=false` и открытых блокерах авто-цикл не стартует обучение.
2. В отчете есть явный статус блокировки readiness.

### F2. Только хорошие сетапы (strict goal)

Проблема: при отсутствии кандидата `>= goal` сейчас выбирается “любой”.

Требование:
1. Ввести строгий режим отбора (по умолчанию включен): тренировать только кандидаты, где `net_return_pct >= goal`.
2. Если таких кандидатов нет, итерация завершается статусом `no_goal_candidate`, без train/oos.
3. Добавить явный счетчик пропусков по причине отсутствия goal-кандидатов.

Критерий приемки:
1. В strict режиме ни одна итерация не обучается на кандидате ниже цели.
2. В summary отчете видно, сколько итераций пропущено по `no_goal_candidate`.

### F3. Корректное восстановление readiness-state

Проблема: в `finally` всегда принудительно пишется `project_ready=false`.

Требование:
1. Перед запуском читать и сохранять исходное состояние readiness.
2. После завершения (успех/ошибка/прерывание) восстанавливать именно исходное состояние.

Критерий приемки:
1. После авто-цикла readiness совпадает с состоянием до запуска.
2. Побочных изменений readiness без явного запроса нет.

### F4. Надежный парсинг результатов подпроцессов

Проблема: парсится только последняя строка stdout как JSON.

Требование:
1. Парсить stdout устойчиво: искать валидный JSON-объект с конца по строкам.
2. Если JSON не найден, логировать диагностический блок (`stdout_tail`, `stderr_tail`, `returncode`).
3. Не считать шаг успешным без обязательных ключей артефактов.

Критерий приемки:
1. Лишние лог-строки в конце stdout не ломают цикл.
2. Ошибки подпроцесса читаемы в summary отчете.

### F5. Сегментация negative memory

Проблема: общий файл negative-memory смешивает разные окна/режимы.

Требование:
1. Разделить negative-memory минимум по ключу:
   - `symbol`, `timeframe`, `train_start`, `train_end`, `test_day`, `goal_net_return_pct`.
2. Либо хранить единый файл с изолированными секциями по указанному ключу.
3. В summary указывать активный memory-key.

Критерий приемки:
1. Отрицательные сетапы одного окна не блокируют другое окно.
2. В отчете прозрачно видно, из какой секции память применялась.
