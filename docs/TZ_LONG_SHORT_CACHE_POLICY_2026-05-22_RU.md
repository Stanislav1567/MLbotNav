# ТЗ: Разделение LONG/SHORT и обязательная очистка кэша прогона

Дата фиксации: 2026-05-22
Проект: MLbotNav
Статус: ACTIVE (обязательное правило выполнения прогонов)

## 1. Что фиксируем прямо сейчас
1. LONG и SHORT запускаются отдельными прогонами.
2. Перед каждым важным прогоном выполняется очистка рабочей памяти прогона.
3. Данные свечей/истории рынка не удаляются при этой очистке.
4. Для LONG используется `--signal-mode long_only`.
5. Для SHORT используется `--signal-mode short_only`.

## 2. Обязательная очистка перед прогоном
Очистка применяется к рабочему контуру прогона (артефакты и память), чтобы новый run не смешивался со старым:
1. `reports/adaptive`
2. `reports/pipeline`
3. `reports/top_strategy`
4. `reports/training`
5. `reports/stages`
6. `reports/final_review`
7. `reports/inference`
8. `models/pipeline`
9. `models/baseline`
10. `data/meta/negative_memory.jsonl`
11. `data/meta/stage_runtime_state.json`
12. `reports/adaptive/negative_setup_memory.json`

Важно:
1. `data/raw/bybit_ohlcv` (свечи) не удалять.
2. Архивировать старые рабочие артефакты перед очисткой в `reports/_archive/...`.

## 3. Почему это правило обязательное
1. LONG и SHORT имеют разную динамику и разные лучшие параметры.
2. Смешивание памяти прогонов и топов ухудшает качество отбора.
3. Раздельные прогоны дают прозрачный итог по каждой стороне отдельно.

## 4. Текущий рабочий режим (уже внедрено)
1. Разделение по сигналам работает через `signal_mode`:
- `long_only`
- `short_only`
- `both`
2. Для текущего этапа допускаются отдельные последовательные прогоны LONG и SHORT в одном кодовом контуре.

## 5. Что уже реализовано по разделению контуров (обновление 2026-05-23)
Реализовано в `adaptive_auto_train` + `top_strategy`:
1. Контур A (`long_only`): отдельные артефакты и память в:
- `reports/adaptive/long_only/*`
- `reports/top_strategy/long_only/*`
2. Контур B (`short_only`): отдельные артефакты и память в:
- `reports/adaptive/short_only/*`
- `reports/top_strategy/short_only/*`
3. Поддержан явный namespace через `--contour-id` (по умолчанию `auto => signal_mode`).

Что остается общим:
1. Рыночные данные (`data/raw/bybit_ohlcv`).
2. Общий контур full-chain аудита (`reports/table_canon_current`), который не содержит side-specific памяти стратегии.

## 6. Критерии приемки текущего правила
1. Любой важный LONG/SHORT запуск начинается после cache-reset рабочей памяти.
2. В отчете прогона явно указан `signal_mode`.
3. Топ-результат не смешивается с предыдущими несброшенными run-артефактами.
4. Свечные данные сохраняются неизменными.
