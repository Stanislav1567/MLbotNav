# Visual Entry: свежий рабочий протокол target-led калибровки

Дата фиксации: `2026-06-30`.

Статус: `FRESH_TARGET_LED_WORKFLOW_READY_NO_ML`.

## Назначение

Этот документ фиксирует новый порядок работы по visual-entry, чтобы не утонуть в старых слоях, версиях, отчетах и частичных улучшениях.

Главное решение: дальше работа идет не от старых runner-версий и не от общего Optuna-перебора, а от пользовательских целевых входов на графике.

Старые результаты V7/V8/V9/V10/V11, Optuna-паспорта и аудиты не удаляются. Они остаются архивом и источником идей, но не являются очередью задач сами по себе.

## Главная цепочка

Рабочая цепочка теперь такая:

```text
график
-> ручные точки T01..T10
-> тип входа
-> стратегия под тип входа
-> паспорт стратегии
-> entry-only проверка на графике
-> target-lock
-> Optuna только внутри готового паспорта
-> multi-day проверка
-> ML только после отдельного APPROVED_FOR_ML
```

## Источник цели

Источником цели является не старый результат стратегии и не общий бэктест.

Источник цели:

1. Чистый график.
2. Ручная пользовательская точка входа `T##`.
3. Правило исполнения: сигнал формируется на закрытой сигнальной свече, вход LONG выполняется на `open` следующей свечи с учетом slippage.
4. Визуальное подтверждение пользователя.

Каждая точка должна иметь карточку:

```text
target_id: T01
symbol: SOLUSDT
timeframe: 1m
day_utc: YYYY-MM-DD
signal_time_utc: время закрытой сигнальной свечи
entry_time_utc: open следующей свечи
side: long
target_type: один из типов ниже
status: gold / candidate / unclear / rejected
why_good: почему вход нужен глазами
must_have: что обязательно должно быть на графике
invalidates: что делает этот вход неверным
linked_strategy: пока пусто
linked_passport: пока пусто
lock_status: unlocked / locked
```

## Типы входов

Первичная классификация:

1. `DEEP_CAPITULATION_LOW` - глубокий пролив, капитуляция, остановка продаж.
2. `SWING_LOW_RECLAIM` - локальный swing-low и возврат.
3. `SUPPORT_RETEST_LOW` - ретест уровня/поддержки.
4. `TREND_DIP_CONTINUATION` - откат в продолжающемся движении.
5. `BOS_FIBO_RECLAIM` - структура, BOS/CHOCH, FIBO-зона, возврат.
6. `VOLUME_WICK_ABSORPTION` - объем, фитиль, поглощение продажи.
7. `HOT_RECLAIM_SUPPORT` - быстрый reclaim возле поддержки.
8. `UNKNOWN_OR_MIXED` - если точка пока не классифицируется честно.

Нельзя насильно загонять все точки в один тип.

## Как выбирать работу

Нельзя начинать с попытки поймать все `T01..T10` одной стратегией.

Правильный порядок:

1. Выбрать один день.
2. Зафиксировать `T01..T10`.
3. Разложить точки по типам.
4. Выбрать один самый понятный тип.
5. Взять 2-4 похожие точки этого типа.
6. Если тип новый и есть только одна понятная точка, разрешается начать с одной точки, но результат считается гипотезой, а не стратегией.
7. Собрать стратегию только под этот тип.
8. Проверять попадания и ложные входы только в рамках этого типа.

## Паспорт стратегии

Паспорт больше не является просто набором параметров.

Паспорт - это контракт:

1. Какой тип входа ловит.
2. Какие `T##` обязан поймать.
3. Какие зоны обязан не трогать.
4. Какие признаки разрешены.
5. Какие параметры можно крутить.
6. Какие попадания становятся `lock`.
7. Какая потеря считается регрессией.
8. Можно ли передавать паспорт в Optuna.
9. Можно ли вообще думать о ML.

## Правила Optuna

Optuna не ищет смысл с нуля.

Optuna разрешается только после того, как:

1. Есть тип входа.
2. Есть стратегия-кандидат.
3. Есть паспорт.
4. Есть ручные цели.
5. Есть PNG overlay.
6. Видно, что стратегия хотя бы визуально рядом с нужными входами.

Запрещено:

1. Запускать общий Optuna-перебор по всем паспортам без target-led контракта.
2. Подбирать параметры по прибыли до проверки входов.
3. Считать минусовой, шумный или визуально неверный результат кандидатом.
4. Передавать что-либо в ML без отдельного `APPROVED_FOR_ML`.

## Правила ML

ML сейчас запрещен.

ML можно обсуждать только после:

1. Нескольких дней ручных целей.
2. Target-lock ledger.
3. Стабильного покрытия нужных входов.
4. Контроля ложных входов.
5. Отдельного ручного решения `APPROVED_FOR_ML`.

До этого все результаты имеют статус `NO_ML`.

## Регрессия

Новая версия не считается улучшением, если она:

1. Уменьшила false, но потеряла хороший locked-вход.
2. Поймала больше входов, но вернула визуальную кашу.
3. Переехала на старый runner без связи с `T##`.
4. Использовала future return, MFE/MAE, TP/SL или entry-candle OHLCV для выбора входа.

Статус такой версии: `REGRESSION_LOST_TARGET_OR_VISUAL_NOISE_NO_ML`.

## Что должен делать новый чат

Новый чат обязан стартовать так:

1. Прочитать этот документ.
2. Прочитать обязательные файлы из `docs/CALIBRATION_NODE_CURRENT`.
3. Не брать старую хронологию как очередь задач.
4. Не продолжать автоматически V7/V8/V9/V10/V11.
5. Сначала создать или подтвердить `T01..T10` на чистом графике.
6. Затем создать target-led ledger.
7. Затем выбрать один тип входа и работать только с ним до понятного результата.

## Первый рабочий шаг

Первый практический шаг нового чата:

1. Выбрать один день для чистого старта.
2. Подготовить чистый график без старых сигналов.
3. Получить или подтвердить `T01..T10`.
4. Создать `target_ledger` по этим точкам.
5. Разложить точки по типам.
6. Предложить первый кластер из 2-4 похожих входов.

До завершения этого шага не запускать Optuna и не строить ML-экспорт.

## Короткая формула

```text
Сначала фиксируем, куда хотим попасть.
Потом определяем, что это за тип входа.
Потом под этот тип строим стратегию.
Потом защищаем хорошие попадания.
Потом докручиваем параметры.
Потом проверяем дни.
И только потом думаем про ML.
```
