## Как работаем до результата

### 1. Не начинаем со стратегии

Сначала фиксируем, куда хотим попасть глазами.

Нельзя начинать с:

1. старой очереди V7/V8/V9/V10/V11;
2. старой хронологии;
3. общего Optuna-перебора;
4. ML/export;
5. TP/SL или future-return логики.

### 2. Подтверждаем цели

Для каждой точки T## нужно подтвердить:

1. `signal_time_utc` - закрытая сигнальная свеча;
2. `entry_time_utc` - open следующей свечи;
3. `target_type`;
4. почему вход визуально нужен;
5. что делает вход неверным;
6. статус `gold / candidate / unclear / rejected`.

До подтверждения точка не становится lock.

### 3. Берем один тип, не все сразу

Первый тип для проверки: `HOT_RECLAIM_SUPPORT`.

Работаем только с `T07/T08`, пока не станет ясно:

1. это действительно один визуальный тип;
2. какие условия на графике обязательны;
3. какие похожие места нельзя трогать;
4. что считается потерей хорошего входа.

Если выяснится, что T07/T08 разные, кластер разбивается, а не насильно склеивается.

### 4. Пишем паспорт-контракт

Паспорт должен отвечать не “какие параметры покрутить”, а:

1. какой тип входа ловим;
2. какие T## обязаны быть пойманы;
3. какие T## не входят в этот тип;
4. какие зоны запрещено трогать;
5. какие признаки разрешены;
6. какие признаки запрещены;
7. что считается regression;
8. можно ли позже отдавать паспорт в Optuna;
9. можно ли вообще обсуждать ML.

До паспорта Optuna запрещена.

### 5. Проверяем entry-only

Первая машинная проверка после паспорта:

1. только entry-only;
2. сигнал на закрытой свече;
3. LONG на `open` следующей свечи;
4. slippage `5 bps`;
5. PNG overlay;
6. scorer по T##;
7. отчет по hits/missed/false.

Запрещено использовать:

1. future return;
2. TP/SL;
3. MFE/MAE;
4. OHLCV entry-свечи для выбора входа;
5. target label как feature;
6. cooldown-сетки как замену логике входа.

### 6. Фиксируем target-lock

Если entry-only слой попал в хорошие точки, создается lock:

1. какие T## защищены;
2. какой допуск по времени;
3. какой false-шум допустим;
4. какая потеря считается regression.

Новая версия не считается улучшением, если она потеряла хороший locked-вход.

### 7. Только потом расширяем

После одного кластера:

1. проверяем следующий похожий кластер;
2. не смешиваем типы без причины;
3. не собираем широкий union, если отдельные кирпичи шумные;
4. не считаем один удачный день стратегией.

### 8. Optuna и ML

Optuna разрешается только когда есть:

1. подтвержденный тип входа;
2. паспорт-контракт;
3. ручные цели;
4. entry-only PNG/scorer;
5. target-lock;
6. понятный список параметров, которые можно крутить.

ML запрещен до отдельного ручного решения:

```text
APPROVED_FOR_ML
```

До этого все результаты имеют статус `NO_ML`.
