# Fresh Target-Led ML Dataset Ladder

Статус: `FRESH_TARGET_LED_ML_DATASET_LADDER_LOCKED_NO_ML_NO_OPTUNA`.

Назначение файла: зафиксировать порядок работы от ручной разметки к будущему ML так, чтобы не перепрыгивать шаги и не смешивать хорошие входы, плохие входы, паспорта, фичи, scorer, Optuna и ML.

## 0. Жесткие границы

До отдельного ручного статуса `APPROVED_FOR_ML` запрещено:

1. запускать ML-обучение;
2. делать ML-export;
3. делать promotion;
4. считать Optuna заменой ручной разметки;
5. переносить результат в торговую модель;
6. использовать future return, TP/SL, MFE/MAE как признаки выбора входа;
7. использовать `entry_open_price`, `entry + 5 bps` или entry-candle OHLCV как признаки выбора входа.

`entry_open_price` и `entry + 5 bps` нужны только для исполнения, учета и визуального контроля.

Это база разметки, а не разрешение на ML.

## 2. Что ML должен получить потом

ML должен получить не картинки и не “прибыль после входа”, а строки с causal-признаками:

1. `label`: good или bad;
2. `target_type`: тип входа;
3. market context до signal candle;
4. индикаторы до signal candle;
5. структуру low/reclaim/retest до signal candle;
6. volume/wick/range признаки до signal candle;
7. context/evidence блоки, если они проверены и не являются standalone-сигналом.

ML учится отличать условия хорошего входа от условий плохого входа. Она не должна просто запоминать “красивую свечку” и не должна видеть будущее движение цены как причину входа.

## 3. Группы признаков

### 3.1. Базовые признаки для будущего ML

Эти группы можно готовить как признаки, если они считаются только на закрытой signal-свече или раньше:

1. returns/range/volatility до сигнала;
2. range-position: где цена находится внутри локального диапазона;
3. room/path context без использования будущего результата;
4. volume ratio, volume z-score, delta volume;
5. body/wick/candle-pattern до сигнала;
6. RSI;
7. MACD;
8. ADX;
9. Stochastic;
10. support/resistance distance/touches;
11. low/reclaim/retest structure.

### 3.2. Context/evidence, не standalone allow

Эти блоки можно смотреть как контекст, но нельзя включать как одиночную команду “входить”:

1. Fibo;
2. BOS/CHOCH;
3. density/VPOC;
4. divergence;
5. широкие `B015/B017`;
6. `B013`;
7. `B020`.

Причина: аудит показал, что часть этих блоков часто встречается и у плохих входов.

### 3.3. Запрещено как standalone allow сейчас

1. `LOW_ANCHOR_RECLAIM` в текущем виде: `0/16` good;
2. простая сумма блоков `safe_core_hit_count`;
3. правило “чем больше блоков совпало, тем лучше”;
4. любой блок, который дает много bad без узкого паспорта.

## 4. Лестница работы без перепрыгивания

### Шаг 1. Freeze разметки

Готово сейчас:

1. good: `28`;
2. bad: `79`;
3. unlabeled: `0`;
4. стрелочные альтернативы `LA026/LA048/LA057/LA059/LA062` не добавлены как gold, пока нет отдельного zoom/time.

### Шаг 2. Выбрать один тип входа

Следующий тип: `SUPPORT_RETEST_LOW`.

Причина: в текущей базе это лучший первый кандидат по типам:

| target_type | total | good | bad |
|---|---:|---:|---:|
| `SUPPORT_RETEST_LOW` | 25 | 9 | 16 |

Важно: `25` — это не 25 сделок в один день. Это все примеры типа `SUPPORT_RETEST_LOW` внутри общей базы `107`.

### Шаг 3. Review-sheet по одному типу

Для `SUPPORT_RETEST_LOW` нужно сделать один рабочий лист:

1. показать все `9 good`;
2. показать все `16 bad`;
3. на каждом мини-графике показать signal, entry, цену, volume;
4. рядом вывести compact-summary: RSI, MACD, ADX, Stochastic, volume, range-position, low/retest/reclaim, BOS/CHOCH/Fibo context;
5. не делать scorer;
6. не делать ML;
7. не делать Optuna.

Результат шага: пользователь смотрит и говорит `норм / фиксить`, а мы формулируем, чем good отличается от bad.

### Шаг 4. Draft-паспорт только для одного типа

После review-sheet можно писать draft-паспорт `SUPPORT_RETEST_LOW`.

Паспорт должен содержать:

1. что является обязательным условием;
2. что является запретом;
3. какие признаки только context;
4. какие признаки не использовать;
5. какие хорошие входы паспорт обязан сохранить;
6. какие плохие входы паспорт обязан отклонить.

### Шаг 5. Entry-only PNG/scorer

Только после draft-паспорта:

1. построить entry-only PNG;
2. посчитать попадания по good и ложные входы по bad;
3. проверить, что хорошие ручные входы не потерялись;
4. если потерялись хорошие входы, это не улучшение.

### Шаг 6. Multi-day проверка

Только после seed-day visual/scorer:

1. проверить на других днях;
2. не подгонять по одному дню;
3. плохие входы не скрывать;
4. все фейлы фиксировать в отчет.

### Шаг 7. Optuna

Optuna разрешается только внутри готового паспорта и только после visual/scorer.

Optuna не должна:

1. искать входы вместо человека;
2. перепридумывать тип входа;
3. оптимизировать future-profit как причину входа;
4. терять хорошие ручные входы.

### Шаг 8. ML

ML начинается только после отдельного ручного `APPROVED_FOR_ML`.

До этого момента мы готовим:

1. clean labels;
2. feature table;
3. passport audit;
4. multi-day validation;
5. список разрешенных/запрещенных признаков.
