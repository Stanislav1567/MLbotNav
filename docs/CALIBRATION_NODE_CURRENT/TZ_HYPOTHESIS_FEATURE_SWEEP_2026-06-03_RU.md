# ТЗ: последовательный перебор гипотез и фич Optuna/APTuna

Дата UTC: `2026-06-03T12:16:43Z`.

## Статус предыдущего кандидата
Текущий `TOP_EXPERIMENTAL` кандидат ставим на паузу.

Мы не двигаем его в production, не делаем unfreeze и не принимаем `GO`.

## Цель
Проверить, что калибровочный узел штатно перебирает каждую гипотезу/фичу по порядку: от минимума к максимуму, по разрешенным сеткам и параметрам.

Главная цель этого ТЗ — не найти лучший profit-кандидат, а доказать, что перебор работает правильно и результаты фиксируются.

## Источник списка

2. Матрица калибровки: `configs/calibration_full_matrix_v1.yaml`.
3. Инвентаризация слотов: `reports/qa_gate/hypothesis_feature_sweep_inventory_20260603T121643Z.json`.
4. Таблица фиксации: `reports/optuna_catalog/index/hypothesis_feature_sweep_table_20260603T121643Z.csv`.

## Количество слотов
Фиксируем 69 логических слотов:

1. `H000` — контрольная базовая гипотеза без отдельной фичи.
2. `H001-H068` — 68 feature rows из `configs/calibration_full_matrix_v1.yaml`.

По факту матрицы:

2. feature rows: `68`;

Если у слота стоит `calibrate=false`, runtime не запускаем: фиксируем `not_calibrated`, причину и переходим к следующему номеру.

## Правило запуска
Для каждого калибруемого слота:

1. Берем слот строго по порядку: `H000`, затем `H001`, `H002`, и так до `H068`.
2. Калибруем на одном закрытом дне.
3. Применяем найденные параметры на следующем закрытом дне.
4. Проверяем `long_only` и `short_only` как два разных стека.
5. `long_only` и `short_only` не смешиваем, не объединяем в один запуск и не считаем одной стратегией.
6. Внутри каждого стека проверяем сетки `narrow`, `medium`, `wide`.
7. Для каждого запуска фиксируем:
   1. slot id;
   2. block;
   3. feature;
   4. params;
   5. min/max/step/count;
   6. grid preset;
   7. mode;
   8. launcher status;
   9. workers status;
   10. best OOS;
   11. trades;
   12. class;
   13. summary path;
   14. top card path, если есть.

## Разделение long и short стеков
`long_only` и `short_only` — это два разных торговых стека.

Правило нумерации:

1. `H001`, `H002`, `H003` и дальше — это номера слотов фич/гипотез, а не номера long/short.
2. `H001-LONG` — читаемая карточка long-стека внутри слота `H001`.
3. `H001-SHORT` — читаемая карточка short-стека внутри слота `H001`.
4. `H001-SLOT` — общая карточка закрытия слота после long+short.
5. `H002` — следующий слот по списку, а не short от `H001`.

Один слот закрывается только после трех карточек:

1. `Hxxx long_only complete`;
2. `Hxxx short_only complete`;
3. `Hxxx slot complete`.

Внутри `long_only` стек проходит:

1. `narrow long_only`;
2. `medium long_only`;
3. `wide long_only`.

Внутри `short_only` стек проходит:

1. `narrow short_only`;
2. `medium short_only`;
3. `wide short_only`.

После `long_only` формируется отдельная long-карточка.
После `short_only` формируется отдельная short-карточка.
После обеих карточек формируется общая карточка слота.

## Jobs, workers и trials
Jobs/workers и trials не выставляются вручную на каждый маленький запуск.
Их должен выставлять профиль прогона.

Стандартный профиль для текущей цепочки: `SWEEP_PROFILE_STANDARD`.

Параллельность:

1. `ProcessWorkers = 3`;
2. `SearchWorkersPerProcess = 3`;
3. всего search workers = `9`;
4. профиль = `3x3/9`.

Trials и timeout по сеткам:

| grid | OptunaTrials | trials per process | timeout |
|---|---:|---:|---:|
| `narrow` | `60` | `20` | `300 sec` |
| `medium` | `120` | `40` | `600 sec` |
| `wide` | `180` | `60` | `900 sec` |

Один полный калибруемый слот в профиле `STANDARD`:

1. `long narrow`: `60`;
2. `long medium`: `120`;
3. `long wide`: `180`;
4. `short narrow`: `60`;
5. `short medium`: `120`;
6. `short wide`: `180`.

Итого: `720` trials на слот, но это 6 отдельных runtime-запусков под управлением контура.

Будущие профили:

1. `FAST` — быстрый smoke/проверка живости;
2. `STANDARD` — текущий рабочий профиль;
3. `BATTLE` — усиленный профиль для важных слотов;
4. `FULL` — максимально полный профиль, если нужно добивать доказательство.

## Правильная механика контура
Целевая механика:

1. `module_sweep_runner --slot H001 --mode long_only` сам проходит `narrow -> medium -> wide`.
2. `module_sweep_runner --slot H001 --mode short_only` сам проходит `narrow -> medium -> wide`.
3. `module_sweep_runner --slot H001 --finalize` собирает общую карточку слота.

До реализации/использования такого runner-а ручной режим разрешен только как временный пошаговый режим:
запускаем один шаг, фиксируем artifact/table/docs, затем берем следующий номер.

## Правило оценки
Плюс не является целью сам по себе.

Мы смотрим:
1. перебралась ли сетка от min до max;
2. не упали ли worker-ы;
3. сохранились ли artifacts;
4. корректно ли фиксируется лучший, худший, no-candidate и not-calibrated результат;
5. можно ли по таблице понять, какая фича/гипотеза что дала.

## Читаемый боевой формат карточки
Для человека используем не сухую машинную строку, а боевой читаемый формат.

Одна карточка слота должна выглядеть так:

```text
H001 | Доходность за 1 бар

Блок:
Цена / волатильность

Техническое имя:
price_volatility.ret_1

Что калибровали:
return_lookback — окно расчета изменения цены

Диапазон:
min=3, max=30, шаг=3

LONG / H001-LONG:
narrow: -21.0732%, trades=5
medium: -8.6502%, trades=4  <- лучший long
wide:   -30.9244%, trades=13

SHORT / H001-SHORT:
narrow: 0.0000%, trades=0   <- лучший short
medium: -5.4244%, trades=2
wide:   -16.7219%, trades=7

Итог:
Сетки прошли штатно. Кандидата нет. Переход к H002.
```

Обязательные русские поля карточки:

1. номер слота;
2. короткое русское название инструмента/гипотезы в заголовке;
3. тип: фича / гипотеза / контроль;
4. техническое имя;
5. что означает инструмент;
6. что калибровали;
7. диапазон min/max/step;
8. `Hxxx-LONG` с результатами `narrow/medium/wide`;
9. `Hxxx-SHORT` с результатами `narrow/medium/wide`;
10. лучший long;
11. лучший short;
12. итог слота;
13. путь к artifact.

Русские названия фиксируем через словарь проекта `configs/features_block.yaml`.
Полный текущий каталог для карточек: `docs/CALIBRATION_NODE_CURRENT/RU_NAMES_CATALOG_2026-06-03.md`.

Машинная CSV-таблица остается индексом и хранит компактные поля. Читаемая карточка/отчет показывает слот как один блок с двумя дочерними стеками `Hxxx-LONG` и `Hxxx-SHORT`.
