# TZ: паспортная калибровка действий

Дата UTC: `2026-06-22T10:10:00Z`.

Статус: `ACTIVE_RULE`.

## Причина

Старые Optuna-матрицы смешивали разные типы ручек:

1. расчетные параметры фичи;
2. линии и пороги сигнала;
3. гипотезы;
4. ML/runtime thresholds;
5. риск и выходы из сделки.

Из-за этого калибровка могла подбирать не действие инструмента, а смесь параметров, которые относятся к разным уровням системы.

## Новое правило

Каждое действие должно иметь паспорт.

Без паспорта действие нельзя считать активной ручкой калибровки.

## Типы паспортов

1. `ENTRY_FILTER` - разрешает или запрещает вход.
2. `ENTRY_SIGNAL` - формирует сторону входа.
3. `EXIT_SIGNAL` - закрывает уже открытую сделку.
4. `DYNAMIC_EXIT` - двигает TP/SL/trailing/partial exit.
5. `RISK_CONTROL` - управляет размером позиции, SL/TP, timeout, ограничениями риска.
6. `CONTEXT_FILTER` - добавляет рыночный контекст, но не является самостоятельным входом.

## Структура паспорта действия

Каждый паспорт должен фиксировать:

1. `ACTION_ID`.
2. `ACTION_TYPE`.
3. входные данные и таймфрейм.
4. что калибруется.
5. диапазоны min/max/step или values.
6. что не калибруется.
7. LONG/SHORT правило.
8. runtime output.
9. как output используется в backtest.
10. ссылку на активную матрицу.

## Подпаспорт backtest

Backtest больше не должен молча использовать случайные старые правила.

Для каждого действия фиксируется подпаспорт:

1. какие output-колонки действия используются;
2. где действие применяется: entry, exit, dynamic exit, risk;
3. какие выходы являются baseline;
4. какие выходы являются экспериментом;
5. какие параметры не относятся к этому действию.

Текущий baseline выхода:

```text
TP / SL / timeout
```

`-DisableTimeoutExit` остается технической диагностикой, но не baseline.

## Новый конфиг-реестр

Активный реестр:

```text
configs/calibration_action_passports.yaml
```

Он фиксирует:

1. блоки `B001..B006`;
2. русское и техническое имя каждого блока;
3. активные паспорта внутри блока;
4. planned passports внутри блока;
5. разрешенные параметры каждого паспорта;
6. активную passport-action матрицу каждого паспорта;
7. legacy frozen матрицы;
8. baseline exit policy.

Правило:
новые паспорта добавляются сначала в этот один реестр, затем получают паспортный документ и только потом passport-action matrix.

Структура блока:

```yaml
blocks:
  B001:
    block_key: price_volatility
    name_ru: "Цена и волатильность"
    active_passports:
      F001:
        action_id: F001_RET1_ALLOW
        allowed_calibration_params: [...]
```

## Legacy frozen

Старые матрицы не удаляются.

Они остаются только как:

1. исторический артефакт;
2. источник для миграции после ручного разбора;
3. материал для сравнения старого и нового поведения.

Их нельзя использовать как активный baseline новых прогонов без паспортной миграции.

## Технический guard

В `src/mlbotnav/optuna_space.py` добавлен `passport_mode`.

Если матрица включает:

```yaml
passport_mode:
  enabled: true
```

то любые параметры вне `allowed_calibration_params` вызывают ошибку компиляции Optuna-space.

Дополнение 2026-06-22:
passport matrix должна сверяться с главным registry:

```yaml
passport_mode:
  registry_path: configs/calibration_action_passports.yaml
  registry_block_id: B001
  registry_passport_id: F001
```

Compile падает, если:

1. блока нет в registry;
2. паспорта нет в блоке;
3. `action_id` не совпадает;
4. `allowed_calibration_params` не совпадают;
5. `active_matrix_path` в registry не совпадает с фактической матрицей.

## Следующий порядок работы

1. Закончить F001 как первый активный паспорт.
2. Для F002 создать паспорт до любой калибровки.
3. Для выходов создать отдельные паспорта `EXIT_SIGNAL` / `DYNAMIC_EXIT`, не смешивая их с F001.
4. Каждую старую строку переносить в новый конфиг только после паспорта.
