# ТЗ: Optuma bridge current

Дата UTC: `2026-06-04T06:55:00Z`.

## Смысл

Работаем через текущий калибровочный контур `Optuma`.

Старый конфиг с min/max используем как опорный мост:

`configs/calibration_feature_hypothesis_draft.yaml`

Текущую матрицу и статус используем как active current:

1. `configs/calibration_full_matrix_v1.yaml`
2. `docs/CALIBRATION_NODE_CURRENT/CURRENT_STATUS_RU.md`
3. `docs/CALIBRATION_NODE_CURRENT/CURRENT_BLOCK_PARAM_RUNTIME_AUDIT_2026-06-04_RU.md`

## Правило

Не запускаем блок вслепую.

Перед прогоном по каждому блоку фиксируем:

1. название блока;
2. какие фичи входят;
3. какие параметры калибруются;
4. `min / max / step`;
5. что реально подключено в коде;
6. что fixed и не калибруется;
7. что нужно починить до прогона.

## Порядок работы

1. Берем следующий блок по current-списку.
2. Сверяем старый конфиг с текущей матрицей.
3. Сверяем параметры с кодом: параметр должен реально влиять на формулу.
4. Если параметр записан, но не работает, фиксируем дефект и ремонтируем.
5. После ремонта запускаем калибровку блока.
6. `long_only` прогоняем отдельно.
7. `short_only` прогоняем отдельно.
8. В каждом направлении проверяем `narrow / medium / wide`.
9. После запуска читаем логи и artifacts.
10. Фиксируем итог блока: что работало, что не работало, лучший результат, ошибки, следующий шаг.

## Критерий закрытия блока

Блок закрыт только если:

1. параметры min/max реально применяются;
2. long и short прошли отдельно;
3. сетки `narrow / medium / wide` проверены;
4. результат сохранен в reports;
5. найденные расхождения либо исправлены, либо явно помечены как fixed/deferred;
6. итог записан в current status.

## Что не делаем

1. Не берем задачи из старой хронологии.
2. Не запускаем production.
3. Не делаем unfreeze.
4. Не считаем plus-кандидата готовым к production без отдельного forward/GO.
5. Не продолжаем следующий блок, если текущий блок не закрыт или не помечен как deferred.

## Первый ремонт перед новым блоковым прогоном

Закрыть разрыв `calibration_params`:

1. сохранить выбранные Optuna параметры после search;
2. передать их в `pipeline_train_eval`;
3. сохранить в train report и model payload;
4. применить в `oos_evaluate`;
5. применить в финальном `run_prob_backtest`.

После этого чинить параметры, которые записаны в YAML, но не работают в формуле:

1. `volume_flow`: `vol_chg`, `delta_volume`, `obv_slope_5`;
2. `density_profile`: `density_vpoc_drift_20`;
3. `structure_ta`: `retest_flag`, `false_breakout_flag`, FIBO;
4. `pattern`: фигурные паттерны и связка `pattern + structure_ta + volume_flow`.
