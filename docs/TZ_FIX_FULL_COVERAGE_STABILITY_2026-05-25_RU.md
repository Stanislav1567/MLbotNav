# TZ: Стабилизация Full-Coverage Прогона (1D train -> 1D test)

Дата: 2026-05-25  
Контур: `run_hypotheses_full_coverage_1d1d.ps1`  
Режим: раздельные прогоны `long_only` и `short_only` (без смешивания)

## 1. Цель
Устранить ложные/технические падения full-coverage прогона и сделать итог детерминированным:
1. Ошибки должны отражать реальную бизнес-причину, а не артефакт пайплайна.
2. Для каждого шага гипотезы должны сохраняться полные диагностические ссылки.
3. File-lock на артефактах (`.xlsx`) не должен ломать цепочку.
4. История изменений и решений должна фиксироваться в едином журнале хронологии.

## 2. Фактические проблемы (по прогону 2026-05-25)
1. `train_failed` с потерей `summary_path` в coverage-отчете.
2. `table_convergence_5plus` падал из-за отсутствующих файлов chain-а после ошибок упаковки.
3. `PermissionError [WinError 32]` на `reports/table_canon_current/data/readable_tables_ru.xlsx`.
4. В блоке `catch` часть диагностической информации зануляется, что усложняет RCA.

## 3. Область работ
1. `run_hypotheses_full_coverage_1d1d.ps1`
2. `src/mlbotnav/table_canon_pack.py`
3. Документация runbook + хронология изменений

## 4. План исправления
### Этап A. Надежность оркестратора (PowerShell)
1. После каждого критичного шага (`adaptive_auto_train`, `table_canon_pack`, `table_convergence_5plus`, `tz_gate_runner`, `p72_freeze_ready_check`) добавить явную проверку кода возврата и fail с точным сообщением шага.
2. В `catch` сохранять `summary_path`, `oos_report`, `convergence_report`, `gate_report`, `freeze_report`, если они уже были получены до ошибки.
3. Добавить в coverage-row поля `failed_stage`, `exit_code`, `stderr_tail` (если доступно), чтобы RCA был самодостаточным без ручного поиска.

### Этап B. File-lock устойчивость (Python)
1. В `table_canon_pack.py` обернуть очистку `run_dir` в retry-механику:
   1. До `N=5` попыток.
   2. Backoff: `0.5s -> 1s -> 2s -> 3s -> 5s`.
   3. В лог/JSON писать факт retry и итог.
2. Если lock не снят, возвращать структурированный `FAIL` с кодом причины `file_locked` и путем файла.
3. Не оставлять частично очищенный каталог без метки состояния.

### Этап C. Консистентность отчетов
1. В coverage-итоге различать статусы:
   1. `MODEL_FAIL` (ошибка train/oos).
   2. `CHAIN_FAIL` (ошибка table-canon/audit chain).
   3. `GATE_FAIL` (гейт не PASS при валидном chain).
2. Убрать двусмысленность, когда все выглядит как один `FAIL`.

### Этап D. Документация и процесс
1. Обновить run-команды в runbook для раздельного запуска `long_only`/`short_only`.
2. Зафиксировать правило: перед прогоном закрывать Excel/просмотрщики артефактов из `reports/table_canon_current/data`.
3. Ввести обязательную запись в хронологию после каждого инцидента/фикса.

## 5. Критерии приемки (AC)
1. `long_only` и `short_only` запускаются отдельными командами и не смешиваются.
2. При искусственном lock файла прогон завершает шаг с понятной причиной `file_locked`, а не «немым» падением.
3. В coverage JSON для каждого `FAIL` есть:
   1. `failed_stage`
   2. `error`
   3. диагностические пути и/или `stderr_tail`
4. После снятия lock повторный прогон на тех же датах проходит без ложных chain-ошибок.
5. Запись в хронологию добавлена в день изменения.
