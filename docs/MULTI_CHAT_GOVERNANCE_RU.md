# MULTI-CHAT GOVERNANCE (Жесткий регламент)

Дата: 2026-05-25  
Статус: ACTIVE  
Область: весь репозиторий `MLbotNav`

## 1. Единая точка правды (Single Source of Truth)
1. Хронология всех действий: `docs/CHANGELOG_CHRONOLOGY_RU.md` (append-only).
2. Активный план фиксов: `docs/TZ_FIX_FULL_COVERAGE_STABILITY_2026-05-25_RU.md`.
3. Этот регламент: `docs/MULTI_CHAT_GOVERNANCE_RU.md`.
4. Реестр задач между чатами: `docs/ACTIVE_WORK_ITEMS_RU.md`.

## 2. Жесткие правила для всех чатов
1. Запрещено делать код-фикс без регистрации задачи в `ACTIVE_WORK_ITEMS_RU.md`.
2. Одна задача = один владелец (`owner_chat`), пока задача в статусе `IN_PROGRESS`.
3. Запрещено менять файлы, уже захваченные другой активной задачей.
4. После каждого шага обязательно:
   1. обновить статус в `ACTIVE_WORK_ITEMS_RU.md`,
   2. добавить запись в `docs/CHANGELOG_CHRONOLOGY_RU.md`.
5. Без этих двух действий шаг считается незавершенным.

## 3. Формат статусов задач
1. `PLANNED` — поставлена, но не взята в работу.
2. `IN_PROGRESS` — взята владельцем, идет работа.
3. `BLOCKED` — есть блокер, нужен отдельный фикс.
4. `DONE` — выполнено, есть PASS/FAIL-артефакты.
5. `CANCELLED` — отменено по решению пользователя.

## 4. Процедура handoff между чатами
1. Текущий чат ставит задаче статус `DONE` или `BLOCKED`.
2. В `notes` указывает:
   1. что сделано,
   2. что осталось,
   3. точные пути к отчетам/логам.
3. Следующий чат берет только задачи в `PLANNED`/`BLOCKED`, переводит в `IN_PROGRESS`.

## 5. Политика «не уходить в сторону»
1. Любая новая работа должна ссылаться на пункт ТЗ (`tz_ref`).
2. Если `tz_ref` не указан — задача не запускается.
3. Эксперименты допускаются только как отдельные задачи с явной пометкой `EXPERIMENT`.

## 6. Критерий «без каши»
1. Все решения/изменения и результаты имеют след:
   1. запись в реестре задач,
   2. запись в хронологии,
   3. ссылка на артефакт PASS/FAIL.
2. Если хотя бы одного элемента нет — задача считается «непрозрачной» и возвращается на дооформление.
