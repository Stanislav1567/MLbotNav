# TZ: Строгий План Исполнения Цепочки (без каши)

Дата: 2026-05-25  
Статус: ACTIVE  
Режим: только по пунктам, без пропусков, с обязательным аудитом после каждого шага.

## 0. Жесткие правила
1. Работаем только в репозитории `MLbotNav`.
2. Long и short всегда раздельно до отдельного шага combined.
3. После каждого шага обязательно:
   1. прогон,
   2. аудит,
   3. запись в `docs/CHANGELOG_CHRONOLOGY_RU.md`,
   4. обновление `docs/ACTIVE_WORK_ITEMS_RU.md`.
4. Без этих действий шаг не принимается.

## 1. P1: Канон данных и паритет свечей (Bybit 1:1)
1. Проверить канонические таблицы:
   1. `candles_canonical`
   2. `feature_frame`
   3. `signal_frame`
   4. `execution_trace`
   5. `strategy_summary`
2. Проверить, что временные ряды UTC, шаг 1m, без дублей/дыр.
3. Проверить parity-отчет по дням.

Критерии приемки:
1. `audit_table_chain` = PASS.
2. `candles_parity_report` без fail-дней.
3. CSV и XLSX версии имеют совпадающие столбцы и число строк.

## 2. P2: Табличная сходимость столбцов (CSV/XLSX/SQL-контур)
1. Проверить заголовки и состав столбцов:
   1. OHLCV/служебные поля.
   2. Все фичи из `FEATURE_COLUMNS`.
   3. Поля сигнала и исполнения.
2. Проверить словари столбцов RU/EN.
3. Проверить соответствие `signal_frame` <-> `execution_trace` <-> `strategy_summary`.

Критерии приемки:
1. Нет отсутствующих/лишних столбцов.
2. `run_id` согласован во всех таблицах.
3. Сходимость `net_return`, `trades`, `PnL` с OOS отчетом.

## 3. P3: Калибровка Long-only (1d train -> next 1d test)
1. Прогон по окнам:
   1. `2026-05-17 -> 2026-05-18`
   2. `2026-05-18 -> 2026-05-19`
   3. `2026-05-19 -> 2026-05-20`
2. Режим: только `long_only`.
3. После каждого окна прогонять table chain audit.

Критерии приемки:
1. Все тех-аудиты PASS.
2. Зафиксировано лучшее tradeful окно long.
3. Есть полный набор артефактов (OOS, backtest CSV, график, chain report).

## 4. P4: Калибровка Short-only (1d train -> next 1d test)
1. Прогон по тем же окнам.
2. Режим: только `short_only`.
3. После каждого окна прогонять table chain audit.

Критерии приемки:
1. Все тех-аудиты PASS.
2. Зафиксировано лучшее tradeful окно short.
3. Есть полный набор артефактов (OOS, backtest CSV, график, chain report).

## 5. P5: Gate/Readiness/Operator цепочка
1. Проверить:
   1. `features_block_audit`
   2. `orderbook_source_audit`
   3. `tz_gate_runner`
   4. `p72_freeze_ready_check`
   5. `p24_latest_pass_report`
2. Проверить lock-аудит финального пакета.

Критерии приемки:
1. Все gate-отчеты PASS.
2. Freeze/release режимы согласованы с текущей задачей.
3. Нет конфликтов артефактов между long/short.

## 6. P6: Combined и финальный handoff
1. Запустить combined-проверку после P3+P4.
2. Сформировать краткий итог:
   1. лучший long,
   2. лучший short,
   3. текущие ограничения,
   4. следующий шаг.
3. Обновить реестр активных задач.

Критерии приемки:
1. Combined chain PASS.
2. В хронологии есть итоговая запись с путями к отчетам.
3. Следующий чат может продолжить без потери контекста.

## 7. Порядок выполнения (фиксированный)
1. P1 -> P2 -> P3 -> P4 -> P5 -> P6.
2. Пропуск шагов запрещен.
3. Любой FAIL: фикс в рамках текущего шага, затем повтор шага.

## 8.1 Обязательный блок: Найденные баги/хрупкости пайплайна (фикс в первую очередь)
1. В раннере нет жесткой остановки после возможного падения `table_canon_pack`.
   1. Сейчас после вызова может идти дальше в convergence/gate и давать каскадные ложные FAIL.
   2. Место: `run_hypotheses_full_coverage_1d1d.ps1:163`.
   3. Требование фикса: после шага `table_canon_pack` делать явную проверку exit-code и fail-fast с понятной ошибкой шага.
2. На FAIL зануляется `summary_path`, теряется диагностика шага.
   1. Место: `run_hypotheses_full_coverage_1d1d.ps1:198`.
   2. Требование фикса: сохранять последнее известное `summary_path`, `oos_report`, `convergence_report`, `gate_report`, `freeze_report` в row FAIL.
3. `table_canon_pack` падает на удалении занятого xlsx (`WinError 32`), ломается целостность цепочки.
   1. Место: `src/mlbotnav/table_canon_pack.py:428`.
   2. Требование фикса: retry/backoff на очистку каталога + структурированный `file_locked` FAIL (без «немого» падения).

Критерий приемки блока 8.1:
1. Ложные каскадные FAIL исключены.
2. В FAIL-строке сохраняется диагностический контекст.
3. При file-lock ошибка детерминированная и объяснимая, с отчетом.
