# Аудит причин "нулевых" и "не калибрующихся" прогонов

Дата: 2026-05-22
Проект: MLbotNav

## 1) Что проверено

1. Качество свечей (raw 1m SOLUSDT):
- Период: 2026-04-21 .. 2026-05-21 (31 день)
- В каждом дне ровно 1440 свечей
- Дубликатов timestamp: 0
- Пропусков шага 60 секунд: 0
- Вывод: источник данных корректный, проблема не в свечах.

2. Масштаб фич и фильтров:
- Для train-окна 2026-05-01..2026-05-14:
  - atr14 median ~0.0007266, p95 ~0.0015714
  - |ema_gap| max ~0.00704, p95 ~0.001839
  - future_return p95 ~0.00454
- Критично:
  - фильтр `min_abs_ema_gap=0.05` отсекает 100% данных (|ema_gap| >= 0.05 -> 0 строк)

3. Логика exchange_like и min_expected_move:
- В `exchange_like` порог min_expected_move сравнивается не с future_return, а с proxy:
  - `pred_move_proxy = confidence * atr14`
- На OOF pred_move_proxy:
  - max ~0.00152, p95 ~0.000195
- Следствие:
  - `min_expected_move >= 0.005` почти гарантирует 0 входов

4. Параметры риска/выхода в текущих прогонах:
- stop_loss=0.004 (0.4%)
- take_profit=0.05 (5%)
- hold_bars=20
- На 1m это слишком жесткая асимметрия (TP сильно дальше, чем SL), часто приводит к убыточным/пустым прогонам.

5. Кандидаты и выбор в adaptive loop:
- Была логическая проблема: adaptive читал ограниченный пул кандидатов из search-репорта, из-за этого торговые кандидаты могли не попадать в обучение.

## 2) Что исправлено в коде

1. Включен параллельный поиск кандидатов (workers) в `search_gate_candidate`:
- Добавлен аргумент `--workers`
- Горизонты обрабатываются в ProcessPoolExecutor

2. В `adaptive_auto_train` добавлен аргумент:
- `--search-workers`
- Прокидывается в `search_gate_candidate --workers ...`

3. Устранен перекос в сторону "нулевых" кандидатов:
- В search scoring добавлен штраф для `trades == 0`
- Это уменьшает вероятность выбора "красивого нуля" вместо реального торгового варианта

4. Расширен пул кандидатов для adaptive:
- В search-отчете добавлен `all_candidates_lite` (полный список в легком формате)
- Adaptive теперь берет `all_candidates_lite` в приоритете

5. Ранее исправлено:
- `turbo` больше не режет repeats до 8 (если задано 10, будет 10)

## 3) Итог аудита (главные причины)

Причина A (критично): несоответствие масштаба фильтров и реальных значений фич.
- `min_abs_ema_gap=0.05` для текущего датасета некорректен
- `min_expected_move=0.005..` для exchange_like proxy почти всегда слишком высок

Причина B (критично): параметризация риска/выхода для 1m слишком агрессивная.
- TP=5% при hold=20 и SL=0.4% статистически неустойчиво для частых входов

Причина C (критично): до фикса adaptive мог не видеть полный торговый пул кандидатов.

Причина D (вторичная): цель `goal_net_return_pct=100` очень высокая для недельного OOS,
часто не достижима на первых итерациях и мешает ранней диагностике качества.

## 4) Рекомендованный диапазон для калибровки (LONG/SHORT отдельно)

LONG (старт):
- min_abs_ema_gap: 0.0 .. 0.001
- min_expected_move: 0.0 .. 0.001
- cooldown_bars: 0..3
- take_profit_pct: 0.006 .. 0.02
- stop_loss_pct: 0.003 .. 0.006

SHORT (отдельный контур):
- те же диапазоны, отдельная память и отдельный top

## 5) Проверка после фиксов

Контрольный search (workers=4, long_only, мягкие пороги) показал:
- total_candidates=96
- tradeful_candidates=32 (ненулевые сделки появились)
- pass_candidates=0 (качество пока слабое, но калибровка уже не "мертвая")

Вывод:
- "Не ищется стратегия" ранее было из-за комбинации зажатых порогов + логики отбора.
- После фикса пайплайн стал действительно перебирать торговые кандидаты.
- Следующий шаг: стабильная сетка risk/TP/SL под 1m и раздельные контуры long/short.
