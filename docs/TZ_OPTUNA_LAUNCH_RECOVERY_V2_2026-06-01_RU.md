# TZ: Optuna Launch Recovery V2 (2026-06-01)

Дата: 2026-06-01
Контур: только `Optuna/APTuna` в `MLbotNav`
Ограничение: `ML runtime` не трогаем, `OptunaMlSignalBackend=off`
Источник фактов: `docs/OPTUNA_PRODUCTION_UNFREEZE_POLICY_GATE_2026-06-01_RU.md`, `docs/ACTIVE_WORK_ITEMS_RU.md`, `docs/CHANGELOG_CHRONOLOGY_RU.md`

## 1. Что уже закрыто
1. Этап `13.4.1-13.4.4` завершен: final GO, controlled unfreeze, post-verification PASS, rollback not required (`P1792-P1795`).
2. Блок исправления и пост-аудита `P1821-P1827` завершен.
3. Reset/hotfix вход в recovery (`P1920-P1923`) завершен.
4. Последний подтвержденный статус: `P1990` triage `NO_CANDIDATE`, `P1991` post-audit `PASS`.

## 2. Что делаем сейчас
1. Активная точка входа: `ПУНКТ 7` recovery-плана.
2. Цель текущего этапа: найти валидного кандидата по правилу:
1. `goal_pass=true`
2. `oos_net_return_pct>0`
3. `oos_trades>0`
3. После появления кандидата выполнить `forward-stability`, затем выпустить финальный quality package `GO/NO-GO`.

## 3. Что не делаем сейчас
1. Не возвращаемся к шагам `13.4.x` (они уже закрыты).
2. Не включаем `ML runtime`, `ML-combo`, `ML-signals`.
3. Не запускаем бесконечные remediation-итерации на одном и том же окне без лимита.
4. Не делаем post-audit после каждого микро-шага; audit выполняется на пакет экспериментов.
5. Не расширяем сетки параметров малыми сдвигами без новой гипотезы.

## 4. Жесткие рамки выполнения
1. Максимум `2` пакета экспериментов в текущем заходе.
2. В каждом пакете не более `3` гипотез `single-change`.
3. Проверка переносимости: использовать `3` разных окна, а не одно окно.
4. После пакета B обязательно выпускать финальное решение (`GO` или `NO_GO`) без продления цикла.

## 5. Пошаговый маршрут (старт -> финиш)
1. Шаг 1. Старт из состояния после `P1991`.
2. Шаг 2. Checkpoint A: зафиксировать 3 окна, 3 гипотезы, состав пакета A.
3. Шаг 3. Пакет A: `long/short` прогоны -> единый triage -> один post-audit.
4. Шаг 4. Checkpoint B:
1. Если кандидат найден, перейти к шагу 5.
2. Если кандидат не найден, перейти к шагу 6.
5. Шаг 5. Forward-stability на 2 следующих окнах -> финальный quality package `GO/NO-GO` -> финиш.
6. Шаг 6. Пакет B (последний): `long/short` прогоны -> единый triage -> один post-audit.
7. Шаг 7. Checkpoint C (обязательное закрытие):
1. Если кандидат найден, выпуск `GO` package.
2. Если кандидат не найден, выпуск `NO_GO` package с RCA и перечнем доработок для следующей версии.
8. Шаг 8. Финиш: решение выпущено, цикл закрыт, новые `P19xx` без нового ТЗ не открываются.

## 6. Definition of Done
1. Обязательный минимум для закрытия этого ТЗ:
1. Выпущен один финальный документ решения (`GO` или `NO_GO`).
2. Приложены артефакты triage/audit по пакетам A/B.
3. Зафиксирован результат forward-stability или формальная причина `NO_GO`.
2. Отсутствие кандидата после пакета B считается валидным завершением через `NO_GO`, а не незавершенной работой.
